# Internationalization (i18n)

goloom uses a single locale catalog per language under [`locales/`](../locales/). The same files feed the web UI and API error messages so translators only maintain one place per language.

## Adding or updating a language

1. Copy [`locales/en.json`](../locales/en.json) to `locales/<code>.json` (for example `de.json` for German).
2. Translate the `ui` object (user interface) and the `api` object (HTTP error bodies).
3. Register the language code in [`frontend/src/i18n/index.ts`](../frontend/src/i18n/index.ts) (`supportedLanguages`) and in [`internal/i18n/catalog.go`](../internal/i18n/catalog.go) if the backend should serve that locale.

Use [BCP 47](https://www.rfc-editor.org/rfc/rfc5646) language tags (`en`, `de`, `fr`, …). Regional variants are fine (`pt-BR`); the matcher uses the primary subtag when a full tag is missing.

## UI strings (React)

- Library: [i18next](https://www.i18next.com/) + [react-i18next](https://react.i18next.com/).
- Keys live under `ui` in the JSON files; the app loads only the `ui` subtree as the `translation` namespace.
- In components:

```tsx
import { useTranslation } from 'react-i18next'

function Example() {
  const { t } = useTranslation()
  return <button>{t('common.save')}</button>
}
```

- Prefer nested keys (`nav.home`, `auth.signIn`) over long flat names.
- User-chosen language is stored in browser settings (`goloom-ui-settings` → `ui.language`) and sent to the API as `Accept-Language`.
- Do not hardcode user-visible English in new UI code; add a key to `locales/en.json` and mirror it in other language files (English fallback is used for missing keys).

## API error messages (Go)

- Keys live under `api` in the same JSON files.
- Handlers should use `a.writeError(w, r, "invalid_json_body", http.StatusBadRequest)` (see [`api/http.go`](../api/http.go)) instead of literal `http.Error` text when the message is user-facing.
- Clients may send `Accept-Language: de` (the UI does this automatically). Unknown languages fall back to English.
- Keys use `snake_case` and stay stable; changing a key is a breaking change for API clients that map errors.

## Mapping server errors in the UI

When the backend still returns English text (or for third-party tools), [`frontend/src/i18n/translateApiError.ts`](../frontend/src/i18n/translateApiError.ts) maps exact English bodies to `api.*` keys. Add an entry there when you introduce a new API message before all handlers use `writeError`.

## Checklist for contributors

- [ ] New UI copy → key in `locales/en.json` under `ui`
- [ ] Same key in every other `locales/*.json` you support (or leave untranslated for English fallback)
- [ ] New API message → key in `locales/en.json` under `api` + use `writeError` in Go
- [ ] Optional: entry in `translateApiError.ts` for legacy/plain-text responses
- [ ] Run `make build` and `cd frontend && pnpm run build`
- [ ] Docker image build copies `locales/` into the frontend stage (see `TestDockerfileCopiesLocalesBeforeFrontendBuild` in `internal/webui/docker_frontend_build_test.go`)

## Testing another language

1. Open **Settings** → **Language** and pick a locale.
2. Reload if some cached labels do not update (most views react immediately).
3. API errors from the server should match the selected language when the handler uses `writeError`.
